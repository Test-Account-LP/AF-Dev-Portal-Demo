Path 1

Build a dApp on Algorand.

You're building something with a frontend. Users will connect wallets, sign transactions, and interact with the chain — even if they don't know that's what they're doing. This is the path for that.

Algorand dApp development works with the tools you already know — TypeScript, React, Python backends — and adds a wallet connection layer, an optional smart contract layer, and a transaction model that won't surprise you.

Required for everything
Start with the Core
TypeScript SDK Python SDK REST API reference Indexer API reference
The SDKs and APIs are your foundation.

Before you think about wallets or contracts, you need a way to talk to the chain. The Algorand TypeScript and Python SDKs handle transaction construction, account management, signing, and submission. The REST APIs (algod and indexer) sit underneath — use them directly if your stack prefers raw HTTP.

Good to know: You can do a lot from here without ever touching a smart contract. Read wallet balances, query asset holdings, submit payments, create tokens — all native API calls. Most chains need a contract for this. Algorand doesn't.
02 / 05
Get users in
Add wallet connection
use-wallet docs WalletConnect guide Supported wallets
use-wallet handles the connection layer.

use-wallet is the standard wallet connection library for Algorand dApps. It abstracts over every major wallet — Pera, Defly, Exodus — and gives you a consistent interface regardless of what your user has installed.

Add it to any React app in minutes. It handles connection state, account switching, and transaction signing — you handle the rest.

App.tsx 
import { useWallet, WalletProvider } from '@algorand/use-wallet'

function App() {
  const { connect, activeAccount, signTransactions } = useWallet()

  return (
    <button onClick={() => connect()}>
      {activeAccount ? activeAccount.address : 'Connect wallet'}
    </button>
  )
}
  • Supported wallets: Pera, Defly, Exodus, WalletConnect-compatible
  • Connection state: Handles connect, disconnect, and account switching automatically
  • Signing: Unified signing interface — your code doesn't change per wallet
03 / 05
Build your transaction flows
Handle transactions
Fee sponsorship guide MBR management Transaction types Atomic transfers
Transaction construction, fee sponsorship, and MBR management.

Most dApp interactions involve building and submitting transactions. The SDK's TransactionComposer handles this — group transactions, set fees, and submit atomically.

Two things worth knowing upfront that most chains don't surface clearly:

Fee sponsorship: Algorand lets your dApp pay transaction fees on behalf of users. This removes friction for new users who haven't funded their account yet. Set it up once in your backend and it's invisible to your users.
MBR management: Every Algorand account needs a small minimum balance to hold assets. Your dApp needs a strategy for this — either fund it upfront or communicate it clearly. Fee sponsorship can cover this too.
sponsor.ts 
const composer = client.newGroup()

// Sponsor covers the MBR for the user
composer.addPayment({
  sender: sponsorAccount,
  receiver: userAccount,
  amount: AlgoAmount.MicroAlgo(100000),
})

// User sends the asset transfer
composer.addAssetTransfer({
  sender: userAccount,
  receiver: recipient,
  assetId: assetId,
  amount: 100n,
})

await composer.execute()

The Algorand REST APIs — algod for submitting transactions and reading current state, indexer for querying historical data — are your primary interface. Any service that can make an HTTP call can integrate Algorand payments without a blockchain SDK.

If you prefer native library support, the Python and TypeScript SDKs wrap the REST layer with typed interfaces and helper methods.

02 / 05
Start simple
Send your first payment
Transaction types USDC on Algorand Fee structure
A payment is one API call.

Stablecoin settlement on Algorand is a native asset transfer. No contract interaction, no ABI, no gas estimation. Fixed fee — currently 0.001 ALGO per transaction, regardless of amount.

payment.py 
from algosdk.transaction import AssetTransferTxn

# Send USDC — one transaction, fixed fee, settled in ~3 seconds
txn = AssetTransferTxn(
    sender=sender_address,
    receiver=recipient_address,
    amt=10_000_000,  # 10 USDC (6 decimal places)
    index=USDC_ASSET_ID,
    sp=client.suggested_params()
)

signed = txn.sign(private_key)
txid = client.send_transaction(signed)

result = wait_for_confirmation(client, txid)
print(f"Settled in round {result['confirmed-round']}")
  • Fee: 0.001 ALGO (~$0.0002) — fixed, always, regardless of amount
  • Finality: ~2.9 seconds — not probabilistic confirmation, actual finality
  • Uptime: 100% since 2019 — no rollbacks, no forks, no surprises in production

Integration patterns
Wire into your existing stack
REST API integration guide Atomic transaction groups ISO 20022 mapping
Your Kafka stays. Your ledger stays. Algorand adds a settlement layer.

The most common integration pattern is a sidecar service — a lightweight Algorand client that sits alongside your existing infrastructure and handles chain interaction. Your core systems stay unchanged.

The pattern: Kafka event → Algorand sidecar → algod REST API → confirmed settlement → Kafka event back. No changes to your core payment ledger. No changes to your compliance engine. One new service, clearly scoped.

For ISO 20022 teams: Algorand's transaction model maps cleanly. Asset transfers are straightforward equivalents to payment messages. Atomic transaction groups handle multi-leg settlements without the coordination overhead you'd expect.

  • Atomic groups: Up to 16 transactions that succeed or fail together — native to the protocol, no oracle, no coordination contract
  • Multi-leg settlement: Group a debit and credit in the same atomic transaction — either both go through or neither does
  • Event confirmation: Poll for confirmation or use webhooks — under 3 seconds either way

Path 3

Build institutional infrastructure on Algorand.

This path is for engineers building custody systems, institutional wallets, B2B settlement infrastructure, and compliance-grade signing flows. The assumption here is that you're not building a consumer product — you're building the infrastructure other products run on.

Algorand is designed to fit into this. Key management is decoupled. The REST layer speaks standard HTTP. The signing model supports multisig, rekeying, and hardware modules without workarounds.

Vault xHD wallets HSM / PKCS#11 MPC native multisig rekeying passkey auth
01 / 05
Start here
Understand the security model
Account model reference Rekeying guide Multisig accounts LogicSig reference
Authorization is enforced at the account level, chain-wide.

Every action on Algorand is authorized by a cryptographic signature tied to an account. There's no shared state, no global admin key, no contract owner pattern that can be exploited across accounts.

  • Rekeying: Any Algorand account can be rekeyed — signing authority transferred to a new key without changing the public address. Native to the protocol, not a contract hack. Useful for key rotation without address migration.
  • Multisig: Native M-of-N multisig at the protocol level. No smart contract required. Thresholds set at account creation. Signatures collected off-chain and submitted as a single transaction.
  • LogicSigs: Programmable signing conditions at the account level. Used for derived accounts, fee delegation, and custom authorization flows.

// Go — custom signing function with HSM
func hsmSign(txn types.Transaction) ([]byte, error) {
    txBytes := msgpack.Encode(txn)
    signature, err := hsm.Sign(keyHandle, txBytes)
    if err != nil {
        return nil, err
    }
    return signature, nil
}

// Pass your signing function to the client
client := algorand.NewClientWithSigner(algodClient, hsmSign)

Transaction signing at scale
Build signing infrastructure
Multisig guide MPC integration reference Rekeying for key rotation
MPC, multisig, and threshold signing — all native.

Algorand's signing model is built for institutional use without workarounds.

multisig.py 
from algosdk.transaction import Multisig

# Create a 2-of-3 multisig account — no contract required
msig = Multisig(
    version=1,
    threshold=2,
    addresses=[key1, key2, key3]
)

print(f"Multisig address: {msig.address()}")
  • Native multisig: M-of-N at the protocol level. Signatures collected off-chain, submitted as one transaction. No contract, no oracle.
  • MPC-compatible: Algorand uses Ed25519 — compatible with standard MPC protocols. Your existing MPC generates the signature; the SDK submits it.
  • Rekeying for rotation: Rotate signing keys without changing the public address your counterparties know. On-chain address stays constant. Audit trail preserved on-chain.

The LTN test suite — validate custody behavior end to end.

Before anything touches mainnet, Algorand's long-term test network (LTN) gives you a persistent, stable environment that mirrors mainnet behavior exactly. Unlike testnet, LTN isn't reset. State persists.

You can run full end-to-end custody validation — key rotation workflows, multisig signing flows, rekeying operations — in an environment that behaves exactly like production. When it works on LTN, it works on mainnet.

Why LTN matters for institutional teams: Testnet resets periodically — you can't run long-lived custody workflows on it. LTN is persistent. Your test accounts, test transactions, and test state stay put between sessions. Run your compliance validation against a real environment, not a toy.
